## 1. Цель проекта

Разработать веб-сервис, который по адресу Flux-кошелька (ZelID / Bitcoin P2PKH / Ethereum) отображает:
- сводку по всему «флоту» нод владельца;
- параметры счёта (баланс, USD-эквивалент, оценочный доход);
- детальные параметры каждой ноды (IP, tier, ранг, uptime, бенчмарк, версия FluxOS, хостящиеся приложения и т.д.);

и параллельно — **Telegram-бот**, отправляющий уведомления о критичных событиях и отвечающий на команды запроса статуса.

Функционально сервис должен быть **близок к оригиналу Fluxnode**, но с **собственным современным дизайном в стиле Flux** (см. §7) и **self-hosted** на нашем сервере.

**Принципиально:** сервис **публичный, без регистрации**. Ввёл кошелёк → видишь данные. Telegram-привязка делается через бота отдельно и не требует учётной записи на сайте.

### 3.2. Стек (зафиксирован)

| № | Компонент | Технология | Назначение |
|---|---|---|---|
| 1 | `web/` | React 18 + Vite + TypeScript + Tailwind CSS + shadcn/ui | SPA-фронтенд |
| 2 | `api/` | **Rust + Axum** (или `actix-web`, на выбор разработчика — обосновать в ADR) | REST-прокси перед Flux API + бизнес-логика |
| 3 | `worker/` | **Rust + Tokio** | Опрос Flux API по расписанию, расчёт алертов |
| 4 | `bot/` | **Rust + teloxide** (`https://github.com/teloxide/teloxide`) | Telegram-бот: команды, отправка алертов. Может быть отдельным бинарём в том же Cargo workspace |

| 6 | `cache` | Redis 7 + `bb8-redis` / `redis-rs` | Кэш ответов Flux API (TTL 30–60 сек), rate-limit, очередь задач |
| 7 | `proxy` | Nginx | Reverse-proxy, TLS, статика фронта |
| 8 | `infra` | Docker Compose (на старте) → опц. Kubernetes | Развёртывание |

**Cargo workspace** организуем так:

```
fluxradar/
├── Cargo.toml          # workspace
├── crates/
│   ├── api/            # бинарь: HTTP API
│   ├── worker/         # бинарь: polling + расчёт алертов
│   ├── bot/            # бинарь: Telegram-бот (команды + отправка)
│   ├── flux-client/    # библиотека: типизированный клиент Flux API
│   ├── domain/         # библиотека: модели, формулы, общие типы
│   └── storage/        # библиотека: репозитории PG + Redis
└── ...
```

Разделение `api` / `worker` / `bot` на три бинаря позволяет масштабировать их независимо и проще писать тесты.

### 3.3. Почему свой прокси, а не прямые запросы из браузера к Flux API

1. **CORS** — Flux API не всегда отдаёт нужные заголовки.
2. **Кэш** — у владельца 100+ нод фронт делает 100+ запросов; нужно агрегировать на бэке.
3. **Rate-limit защита** — чтобы один пользователь не положил наш IP в бан Flux API.
4. **Бизнес-логика** — расчёт earnings, агрегации, привязка к ТГ.
5. **Безопасность** — токен ТГ-бота не должен утекать в браузер.

### 3.4. Переиспользование Rust-кода оригинала

В `api/` оригинала уже есть Rust-прокси перед Flux API. План:
1. Сначала прочитать `api/src/` оригинала, понять, какие эндпоинты Flux он проксирует и как.
2. Если лицензия позволяет — взять структуру за основу и расширить (добавить кэш в Redis, агрегации, формулы earnings из `client/src/utils/` оригинала, портированные на Rust).
3. Если лицензия не позволяет переиспользовать — реализуем с нуля по той же логике, ссылаясь только на публичную документацию Flux API.

## 4. Функциональные требования

### 4.1. Главный экран (`/` или `/home`)

**Поле ввода кошелька** + кнопка «Загрузить». Поддерживаемые форматы:
- ZelID (Bitcoin P2PKH, начинается с `1...`)
- Ethereum-адрес (`0x...`)
- (опц.) поиск по IP ноды

После загрузки выводится **дашборд**:

**Блок «Сводка» (header-карточки):**
- Текущая цена FLUX в USD (источник: CoinGecko или Flux API)
- Общее число нод владельца
- Разбивка по tier: Cumulus / Nimbus / Stratus (количества + иконки)
- Баланс кошелька: FLUX + USD-эквивалент
- Оценочный месячный доход (FLUX + USD)
- Оценочный годовой доход (FLUX + USD)
- (опц.) Parallel Assets — холдинги в параллельных активах Flux

**Блок «Spotlight» (3 карточки):**
- **Best Uptime** — нода с самым длинным uptime
- **Highest Ranked** — нода с лучшим рангом в очереди выплат
- **Most Hosted** — нода с максимальным числом приложений

**Блок «Privacy Mode» (тогл):**
Скрывает: адрес кошелька (заменяет на `0x...1234`), IP нод (на `***.***.***.***`), географию.

**Блок «Auto-Refresh»:**
Тогл + выбор интервала (30с / 1мин / 5мин). По умолчанию выключен. Используем эндпоинт `/api/v1/wallet/:address/summary` с проставленным `If-None-Match` для экономии трафика.

**Кнопка «Получать алерты в Telegram»:**
Открывает модалку с инструкцией: «Откройте бота `@ваш_бот`, отправьте команду `/link <ваш-адрес>`». Без какой-либо регистрации на сайте.

### 4.2. Экран «Все ноды» (`/nodes`)

Таблица (рекомендация — **TanStack Table** или **AG Grid Community**) со столбцами:

| Колонка | Источник данных |
|---|---|
| IP | `/daemon/listfluxnodes` |
| Tier (Cumulus / Nimbus / Stratus) | вычисляется по collateral |
| Rank | `/daemon/viewdeterministicfluxnodelist` |
| Status (online / offline / dos) | `/daemon/getfluxnodestatus` |
| Benchmark status (passed / failed) | `/benchmark/getbenchmarks` |
| Uptime | расчёт из `activeSince` |
| Last paid | из истории транзакций |
| Maintenance window | из ответа daemon |
| FluxOS version | запрос к ноде напрямую (или из агрегации сети) |
| Apps count | `/apps/listrunningapps` на ноде |
| Country / Город | `/daemon/getfluxnodegeolocation` |
| Actions | кнопка «детали» |

**Требования к таблице:**
- сортировка по всем колонкам;
- фильтрация (поиск по IP, tier-фильтр, status-фильтр);
- пагинация или виртуальный скролл (если >100 нод);
- экспорт в CSV;
- клик по строке → drawer с деталями ноды (см. §4.3).

### 4.3. Детали ноды (drawer/модалка по клику)

- Полная информация по бенчмарку (EPS, DWS, download / upload speed)
- Список запущенных приложений (имя, образ, ресурсы)
- График uptime за 7 / 30 дней (если есть история — см. §5.4)
- Кнопки «Скопировать IP», «Скопировать команду SSH»

### 4.4. Экран «Гайды» (`/guide`)

Статическая страница со ссылками на:
- официальные гайды Flux (внешние ссылки)
- наши собственные мини-инструкции (стартуем с одной — «Как подписаться на алерты в Telegram»)
- набор полезных команд (copy-to-clipboard блоки)

### 4.5. Демо-режим (`/demo`)

Загружает фиксированный «sample wallet» (захардкоженный в конфиге фронта), чтобы новый пользователь мог посмотреть UI без своего кошелька. **Данные подгружаются реальные** — просто адрес подставляется автоматически.

## 5. Telegram-бот

### 5.1. Команды бота (финальный список)

| Команда | Описание |
|---|---|
| `/start` | Приветствие + инструкция «отправь `/link <адрес>` чтобы получать алерты» |
| `/link <wallet>` | Привязать кошелёк к этому TG-чату. Поддерживается несколько кошельков на один `tg_user_id` |
| `/unlink <wallet>` | Отвязать конкретный кошелёк |
| `/wallets` | Список привязанных к моему чату кошельков |
| `/status` | Краткая сводка по всем моим кошелькам: общее число нод, онлайн/офлайн, баланс |
| `/nodes [wallet]` | Список нод. Без аргумента — по всем привязанным; с аргументом — только по этому кошельку |
| `/settings` | Inline-меню: включить/выключить типы алертов (см. §5.2) |
| `/mute 1h\|6h\|24h` | Временно заглушить все алерты |
| `/unmute` | Снять заглушку досрочно |
| `/help` | Список команд |

**Хранение привязок** — в таблице `telegram_subscriptions` (см. §5.5). Сайт **не имеет** доступа к этому списку — он используется только воркером и ботом.

### 5.2. Типы алертов (финальный набор)

Только то, что было выбрано:

**Группа A — состояние нод:**
1. **Нода ушла в офлайн** (`status != "CONFIRMED"` или нет ответа от ноды > 5 мин подряд)
2. **Нода вернулась в онлайн** (парный алерт к (1) — чтобы пользователь знал, что инцидент закрыт)
3. **Упал бенчмарк** (`/benchmark/getbenchmarks` → `passed: false`)
4. **Бенчмарк восстановился** (парный к (3))
5. **Устарела версия FluxOS** (текущая < latest stable на ≥1 минорной версии)
6. **Устарела версия Benchmark**

**Группа B — кошелёк и выплаты:**
7. **Поступление на кошелёк** (новая входящая транзакция; источник — `/explorer/transactions` с фильтром по адресу, сравниваем с предыдущим тиком)
8. **Выплата получена** (входящая транзакция от Flux coinbase / fluxnode payout — отличаем по структуре vout)

Каждый тип алерта — **независимый тогл** в `/settings`. По умолчанию все включены.

### 5.3. Антиспам

- **Дедупликация:** один и тот же алерт по одной ноде не отправляется повторно в течение **1 часа** (хранится в `alert_dedup` в Redis).
- **Группировка:** если за один polling-тик >5 нод одного кошелька ушли в офлайн — отправляется **одно сообщение** «N нод офлайн (список)», а не N отдельных.
- **Тихие часы (опц.):** в `/settings` можно задать окно «не беспокоить» — например, 23:00–08:00 в TZ пользователя. Алерты типов 1, 3, 7, 8 откладываются до конца окна и отправляются дайджестом. **На MVP — опционально**, можно сделать в Этапе 2.
- **Mute** через `/mute Nh` пишет timestamp в `telegram_subscriptions.muted_until` — все алерты до этой даты пропускаются.

### 5.4. Polling-стратегия

- Базовый цикл: каждые **60 секунд** worker выполняет проход:
  1. Один запрос `/daemon/viewdeterministicfluxnodelist` → кэш в Redis на 60 сек. **Это один запрос вместо N**: список всех ~8000 нод сети.
  2. Один запрос `/benchmark/getbenchmarks` (если поддерживается на сетевом уровне) или по выборке.
  3. Для каждого уникального адреса кошелька из `telegram_subscriptions`:
     - фильтруем список нод по `payment_address`;
     - сравниваем со снапшотом предыдущего тика (`node_status_snapshots`);
     - формируем дельту → события → пайплайн алертов.
- Транзакции для алертов B: отдельный, **более частый** цикл (например, 30 сек) с эндпоинтом `/explorer/transactions?address=...` (или поднимаем подписку на новые блоки, если будет реализовано).
- **Backoff:** если Flux API вернул 429 / 5xx — экспоненциальная задержка + jitter.

### 5.5. Схема БД для бота

```sql
-- Привязки TG ↔ кошелёк
CREATE TABLE telegram_subscriptions (
    id              BIGSERIAL PRIMARY KEY,
    tg_chat_id      BIGINT NOT NULL,
    tg_user_id      BIGINT NOT NULL,
    wallet_address  TEXT NOT NULL,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    muted_until     TIMESTAMPTZ,
    settings_json   JSONB NOT NULL DEFAULT '{}'::jsonb,  -- какие типы алертов включены
    UNIQUE (tg_chat_id, wallet_address)
);
CREATE INDEX idx_telegram_subscriptions_wallet ON telegram_subscriptions(wallet_address);

-- Снапшоты статусов нод для дельта-вычисления
CREATE TABLE node_status_snapshots (
    node_ip         TEXT PRIMARY KEY,
    wallet_address  TEXT NOT NULL,
    status          TEXT NOT NULL,         -- CONFIRMED / EXPIRED / DOS / OFFLINE
    benchmark_ok    BOOLEAN NOT NULL,
    flux_os_version TEXT,
    bench_version   TEXT,
    last_seen_at    TIMESTAMPTZ NOT NULL,
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_node_status_wallet ON node_status_snapshots(wallet_address);

-- История отправленных алертов (для аудита и дедупликации)
CREATE TABLE alerts_history (
    id              BIGSERIAL PRIMARY KEY,
    tg_chat_id      BIGINT NOT NULL,
    wallet_address  TEXT NOT NULL,
    alert_type      TEXT NOT NULL,         -- 'node_offline' / 'benchmark_failed' / 'incoming_tx' / ...
    subject         TEXT,                  -- IP ноды / txid
    payload_json    JSONB,
    sent_at         TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_alerts_history_subject ON alerts_history(alert_type, subject, sent_at DESC);

-- Последняя обработанная транзакция на кошелёк (для алертов B)
CREATE TABLE wallet_tx_cursor (
    wallet_address  TEXT PRIMARY KEY,
    last_txid       TEXT,
    last_block      BIGINT,
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

## 7. Дизайн

### 7.1. Стиль

**«Flux-стиль, современный»:**
- тёмная тема по умолчанию + светлая опция;
- акценты в фирменных цветах Flux: глубокий синий `#2B61D1` (или похожий — сверить с их сайтом), бирюзовый/циан акцент;
- **glassmorphism / subtle gradients** — карточки с лёгким blur-фоном;
- **много пространства**, крупная типографика для чисел (баланс, число нод);
- **анимированные счётчики** при загрузке (число «накручивается» от 0 до значения);
- скругления `rounded-2xl`, мягкие тени;
- иконки — Lucide или Tabler.

### 7.2. Что НЕ делать

- Не делать «AI-generic»: фиолетовый градиент + emoji + Inter — это шаблон, выглядит дёшево.
- Не перегружать дашборд графиками — в первой версии достаточно карточек + одной таблицы.
- Не использовать логотипы/иконки оригинального Fluxnode.

### 7.3. Дизайн-система

- **Шрифт:** один из — Geist, JetBrains Sans, Manrope, Space Grotesk
- **Моноширинный для адресов/IP:** JetBrains Mono / Geist Mono

## 8. Endpoints собственного бэкенда

Сервис публичный, авторизация на API не требуется (только rate-limit).

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/api/v1/wallet/:address/summary` | Сводка по кошельку (баланс, число нод, доход) |
| GET | `/api/v1/wallet/:address/nodes` | Список нод владельца |
| GET | `/api/v1/node/:ip/details` | Детали конкретной ноды (бенчмарк, apps, версии) |
| GET | `/api/v1/network/price` | Текущая цена FLUX |
| GET | `/api/v1/network/stats` | Общая статистика сети (для футера / `/demo`) |
| GET | `/api/v1/health` | Liveness probe |
| GET | `/api/v1/ready` | Readiness probe (проверяет PG, Redis, доступность Flux API) |
| GET | `/api/v1/metrics` | Prometheus метрики (защитить basic-auth или IP-allowlist) |

**Кэширование:** все `wallet/*` и `node/*` отвечают с `ETag` + `Cache-Control: max-age=30`. Бэкенд хранит результат в Redis с тем же TTL.

**Rate-limit:** 60 запросов/минуту с IP, плюс 600/час. Реализация — `tower-governor` (если Axum) или собственный middleware на Redis.

## 9. Безопасность

1. **Никогда не запрашивать у пользователя приватные ключи или seed-фразы.** Сайт работает только с публичными адресами.
2. **Telegram Bot Token** — только в `.env` сервера, никогда не в репозитории. В `.env.example` — плейсхолдер.
3. **Rate-limit** на API (см. §8).
4. **Input validation** на адрес кошелька: regex-проверка форматов ZelID / ETH / BTC P2PKH **до** обращения к Flux API. Невалидный адрес → 400, без походов наружу.
5. **CORS:** на проде разрешён только наш домен.
6. **Защита от SSRF в прокси:** если когда-либо принимаем URL от пользователя — белый список доменов; сейчас не принимаем.
7. **HTTPS only**, HSTS включён.
8. **Логи** не должны содержать `tg_user_id` и полные адреса на уровне ERROR в открытом виде. На DEBUG-уровне допустимо.
9. **Защита бота от спама команд:** rate-limit на уровне `tg_user_id` (например, 30 команд/мин).
10. **Webhook для бота** (если используем не long polling) — секрет в URL + проверка `X-Telegram-Bot-Api-Secret-Token`.

## 13. Критерии приёмки MVP

1. Открываю сайт → ввожу публично известный «жирный» кошелёк (например, из rich list) → вижу корректное число нод и баланс, совпадающие с оригинальным Fluxnode (±погрешность кэша 1 мин).

## 14. Открытые вопросы (на согласование)

1. **Лицензия оригинала Fluxnode.** Если AGPL — наш форк/производное тоже AGPL и должен быть открыт. Если просто референс — можно делать с нуля под любой лицензией. **Проверить в первый день работы и зафиксировать в `LICENSE_NOTES.md`.**
2. **Источник цены FLUX.** CoinGecko бесплатный (rate-limit 10–30 req/min) или платный, либо собственный агрегатор с CMC + Binance.
3. **Геолокация нод.** Использовать ту, что отдаёт Flux, или подключать MaxMind/ipinfo для уточнения.
4. **Хранение истории.** Какой период хранения для `node_status_snapshots` и `alerts_history` — 30 / 90 / 365 дней? Влияет на размер БД.
5. **Parallel assets в MVP** — нужны или достаточно в Этапе 2?
6. **Webhook vs long polling для бота.** Webhook требует публичного HTTPS-эндпоинта (он у нас будет), быстрее. Long polling проще на dev. Рекомендация: dev — long polling, prod — webhook.

## 15. Глоссарий

- **ZelID** — публичный адрес кошелька Flux (Bitcoin P2PKH формат, начинается с `1`).
- **Tier (Cumulus / Nimbus / Stratus)** — три уровня нод, отличаются требуемым залогом (collateral) и наградой.
- **Benchmark** — периодическая самодиагностика ноды (EPS, диск, сеть); от результата зависит участие в выплатах.
- **DoS list** — список нод, временно исключённых из сети из-за плохого поведения / недоступности.
- **Maintenance window** — окно обслуживания, в которое нода может ребутнуться без штрафа.
- **EPS / DWS** — Events Per Second / Disk Write Speed, ключевые бенчмарк-метрики.
