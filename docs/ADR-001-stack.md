# ADR-001: Выбор web-фреймворка для `crates/api/` — Axum

- **Статус:** Принято
- **Дата:** 2026-05-25
- **Контекст ТЗ:** §3.2 (стек)
- **Связанные документы:** [LICENSE_NOTES.md](./LICENSE_NOTES.md)

## Контекст

ТЗ фиксирует Rust для бэкенда, но оставляет выбор web-фреймворка за разработчиком: **Axum** или **actix-web** (§3.2), с требованием обосновать выбор в этом ADR.

Решение по лицензии (см. `LICENSE_NOTES.md`) — **писать код с нуля**, оригинал используется только как «референс по фактам». Поэтому выбор фреймворка делается **по техническим достоинствам**, а не из соображений совместимости с чужим кодом.

## Решение

Используем **Axum** для `crates/api/`.

## Обоснование

1. **Единый async-рантайм со всем workspace (главный фактор).** `worker/` (Tokio) и `bot/` (teloxide, тоже на Tokio) живут в экосистеме Tokio. Axum — нативно Tokio, не вносит конкурирующих абстракций рантайма. Весь workspace остаётся на чистом Tokio, общие типы и async-утилиты переиспользуются между крейтами без адаптеров.
2. **Экосистема `tower` / `tower-http`.** Axum построен на `tower`, что даёт готовые middleware-слои под прямые требования ТЗ:
   - rate-limit через `tower-governor` (упомянут в §8 ТЗ как предпочтительный для Axum);
   - CORS через `tower-http::cors` (§9 — на проде только наш домен);
   - структурные JSON-логи через `tower-http::trace` + `tracing` (§6).
3. **Бесшовная интеграция с нашим слоем `storage/`.** `sqlx` и `bb8-redis` — Tokio-native; в Axum они подключаются через extractor `State` без обвязки.
4. **Композиционная модель хендлеров.** Извлечение данных через типизированные extractors (`Path`, `Query`, `State`) хорошо ложится на наш набор эндпоинтов (§8): `/wallet/:address/summary`, `/wallet/:address/nodes`, `/node/:ip/details` и т.д.

actix-web — зрелый и производительный фреймворк, но его actor-наследие и собственный системный слой не дают преимуществ для нашего сценария (тонкий REST-прокси + кэш), при этом добавляют концептуальный вес. Для команды, держащей весь бэкенд на Tokio, Axum проще и однороднее.

## Версия

Берём **актуальную мажорную версию Axum (0.8.x)** и совместимую `tower-http`. Стартуем сразу на современном API (`axum::serve(tokio::net::TcpListener, ...)`), без легаси.

## Последствия

- `crates/api/` пишется с нуля на актуальном Axum; структуру эндпоинтов берём из §8 ТЗ.
- Все HTTP-middleware (CORS, rate-limit, trace, ETag/`Cache-Control`) реализуются как `tower` layers.
- Зависимость от экосистемы Tokio закрепляется во всём workspace — согласуется с `worker/` и `bot/`.
- Решение не зависит от кода оригинала и остаётся в силе вне зависимости от лицензионного статуса Fluxnode.
