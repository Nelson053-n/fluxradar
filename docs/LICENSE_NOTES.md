## Юридическое обоснование

Отсутствие лицензии **не делает код public domain** и не даёт права на использование:

1. **Авторское право по умолчанию** (Berne Convention): произведение защищено с момента создания. Без явной лицензии все права сохраняются за автором.
2. **GitHub Terms of Service** (раздел D, «License Grant to Other Users»): публикация репозитория на GitHub даёт другим пользователям право **просматривать и форкать** репозиторий в пределах GitHub. Это **не** лицензия на копирование кода в сторонний продукт, модификацию вне форка или коммерческое использование.
3. Следовательно: читать код для понимания логики — допустимо; **копировать или портировать его в наш продукт — нет**.

## Практические последствия для нашего проекта

Согласно принятому решению (пишем с нуля, без использования кода оригинала):

### Режим работы с оригиналом: «референс по фактам» (выбор заказчика, 2026-05-25)

Граница проходит по принципу **idea–expression dichotomy**: авторское право защищает *выражение* (конкретный текст кода), но не *идеи, факты и методы*.

**Можно (это не охраняется копирайтом):**
- Узнавать из оригинала **какие эндпоинты Flux API** он использует и в каком порядке — это факты о публичном API (они и так в документации Flux).
- Понимать **смысл формул** earnings / tier-логики (например, из чего складывается оценочный доход) — математический метод не охраняется; охраняется лишь конкретная его запись в коде.
- Перенимать **функциональность и поведение** (что на дашборде, какие колонки) — это и так задано нашим ТЗ.

**Нельзя (это и есть выражение → риск):**
- Копировать или построчно переписывать `.rs` / `.tsx` файлы оригинала.
- «Переводить» их код в свой, держа исходник открытым рядом, — даже с переименованием это производная работа.
- Воспроизводить нетривиальную структуру модулей/функций один-в-один.

**Метод, гарантирующий чистоту:** при взгляде в оригинал выписываем *спецификацию словами* (факты: «для earnings нужны collateral по tier, размер очереди, цена FLUX»), а не куски кода; затем реализуем по этой спецификации и докам Flux API. Корректность сверяем не с чужим кодом, а с публичным результатом — известный rich-list-кошелёк, наши числа должны совпасть с живым сервисом ±погрешность кэша (§13.1 ТЗ).

**Делаем:**
- Реализуем `crates/api/`, формулы earnings, tier-логику и клиент Flux API **самостоятельно**, опираясь на:
  - публичную документацию Flux API: https://docs.runonflux.io/fluxapi
  - OpenAPI-спеку: https://docs.runonflux.io/_bundle/fluxapi.json
  - нашу спецификацию: `docs/TZ_Fluxnode_Service_v1.1.md`
  - факты/идеи из оригинала (в рамках границы выше).
- Корректность формул earnings фиксируем golden-тестами на публично проверяемых значениях (§13.1 ТЗ).

**НЕ делаем:**
- Не копируем и не портируем построчно код `api/` и `client/` оригинала (см. границу выше).
- Не копируем брендинг оригинала (логотип, шрифты, точные цвета их UI) — у нас свой дизайн из `docs/mockup_dashboard.html` (§2 ТЗ).
- Не копируем геймификацию/ачивки в MVP (backlog, Этап 3).

**Лицензия нашего проекта:** так как мы не создаём производное от чужого кода, репозиторий **свободен в выборе лицензии** (любая open-source или проприетарная). Конкретный выбор — отдельный вопрос к заказчику, не блокирует разработку.
